請說明 Spring 的 DI 與 AOP 是什麼？常見注入方式有哪些？你會如何選擇？

觀察重點：是否理解 IoC 容器、依賴注入、AOP proxy，以及 constructor / setter / field injection 的取捨

L

只知道 DI 是注入、AOP 是切面，或只會背 `@Autowired`，但無法說明 Spring 容器與 proxy 的角色。

M

能說明 constructor、setter、field injection 的基本差異，也知道 AOP 可做 logging、transaction 等橫切邏輯。

H

能明確主張 constructor injection 為預設選擇，說出測試性、不可變性、循環依賴、proxy 自呼叫等實務限制。

參考解答

  • DI 是 Dependency Injection，由 Spring IoC 容器負責建立 Bean、管理生命週期，並把相依物件注入進需要它的類別
  • AOP 是 Aspect-Oriented Programming，透過 proxy 在方法呼叫前後加入橫切邏輯，例如 transaction、logging、audit、metrics、權限檢查
  • constructor injection：適合必要依賴，能讓欄位設為 `final`，測試容易，也能在建立物件時暴露缺少依賴的問題，通常是首選
  • setter injection：適合可選依賴或需要後續調整的設定，但物件可能短暫處於依賴不完整狀態
  • field injection：寫法短，但測試不易、隱藏依賴、難以建立不可變物件，正式專案通常不建議作為預設做法
  • AOP 要注意 Spring proxy 限制：同類別自呼叫通常不會觸發 advice，private method 不是典型切點，非 Spring 管理物件也不會被攔截
  • 以此作業來看，Controller 依賴 Repository 或未來的 Service 時，建議改用 constructor injection；交易與日誌等橫切邏輯再考慮 AOP

如果幣別新增與外部匯率查詢要組成同一個業務流程，Transaction 應該怎麼設計？

觀察重點：是否理解 `@Transactional` 邊界、rollback 規則與外部 API 不應被包進長交易

L

只知道要加 `@Transactional`，但不知道應該加在哪一層或 rollback 條件。

M

能說明交易通常放在 Service layer，DB 多步驟操作應同一個交易完成。

H

能指出外部 HTTP 呼叫不宜放在長交易中，會拉長 DB lock 時間；能比較先呼叫外部 API 再交易寫 DB、或用補償機制的取捨。

參考解答

  • `@Transactional` 建議放在 Service 方法，而不是 Controller，讓業務流程與交易邊界一致
  • 多個 DB 寫入或讀寫一致性要求高的操作應在同一交易內，查詢可考慮 `readOnly = true`
  • 外部 API 呼叫可能 timeout 或變慢，不適合包在 DB 交易中，否則會延長連線與 lock 持有時間
  • Checked exception 預設不 rollback，必要時要設定 `rollbackFor`，也要避免同類別自呼叫造成 transaction proxy 失效
  • 此作業規模可先抽 Service 並讓 DB 寫入由交易包住；外部查詢失敗時回傳明確錯誤即可，不需過度導入分散式交易

如果使用 OKHttp3 HttpClient 呼叫 CoinDesk API，應該注意哪些設計與資源管理問題？

L

M

H

能比較 singleton `OkHttpClient`、connection pool、interceptor、MockWebServer、錯誤映射與此作業規模下的封裝方式。

參考解答

  • 可用 interceptor 放 logging、header、trace id；測試可用 OkHttp `MockWebServer` 模擬 CoinDesk

請說明 `java.util.stream` 的優點，並示範如何用 Stream 對 `List<Policy>` 依保單號 `policyNum` 分組、攤平成集合，以及用 `Collectors.toMap` 保留原始順序。

觀察重點：是否理解 Stream pipeline、`map`/`flatMap`/`groupingBy`/`toMap`、重複 key 與順序保留

L

只知道 Stream 可做迴圈簡化，或只會背 `map` 名稱，但無法寫出分組或 `toMap` 實作。

M

能寫出 `groupingBy`、`map`、`flatMap` 的基本用法，但對重複 key、順序保留、null 與可讀性取捨掌握不足。

H

能完整說明 Stream 適用情境與限制，能用 `LinkedHashMap` 保留順序、處理 duplicate key，並能判斷何時傳統迴圈更清楚。

參考解答

  • Stream 優點是能以宣告式 pipeline 表達轉換、過濾、分組、彙總，適合集合資料處理，也容易串接 `filter`、`map`、`collect`
  • 依保單號分組可使用：`policies.stream().collect(Collectors.groupingBy(Policy::getPolicyNum))`，回傳 `Map<String, List<Policy>>`
  • 若要讓分組結果保留第一次遇到的保單號順序，可使用：`Collectors.groupingBy(Policy::getPolicyNum, LinkedHashMap::new, Collectors.toList())`
  • `map` 是一對一轉換，例如 `policies.stream().map(Policy::getPolicyNum)`；`flatMap` 是一對多後攤平，例如 `groups.stream().flatMap(group -> group.getPolicies().stream())`
  • `toMap` 若要保留原始順序，需指定 map supplier：`Collectors.toMap(Policy::getPolicyNum, Function.identity(), (first, second) -> first, LinkedHashMap::new)`
  • `toMap` 要注意 duplicate key 會拋 `IllegalStateException`，因此要明確提供 merge function；也要注意 key/value 是否可能為 null
  • 工程判斷上，Stream 適合中短鏈路資料轉換；若邏輯分支多、副作用多、debug 困難，傳統迴圈或拆小方法可能更清楚

Spring 中 middleware、Filter、Interceptor 與 AOP 的區別是什麼？你會如何選擇使用時機？

觀察重點：是否理解請求生命週期、Servlet Filter、Spring MVC Interceptor、AOP proxy 與橫切關注點邊界

L

只知道它們都可以做攔截或前後處理，但說不出執行位置、作用範圍與限制。

M

能分辨 Filter 在 Servlet 層、Interceptor 在 Spring MVC 層、AOP 在 Bean method 層，但選型取捨較粗略。

H

能依情境選擇：安全與 request wrapping 用 Filter、Controller 前後處理用 Interceptor、service 橫切邏輯用 AOP，並能指出 proxy、自呼叫與順序問題。

參考解答

  • `middleware` 在 Spring 裡通常是泛稱，可能指 HTTP request pipeline 中的 Filter / Interceptor，也可能泛指橫切處理元件，面試時要先釐清語境
  • `Filter` 是 Servlet 規格層級，進入 Spring MVC 前就會執行，適合 CORS、安全驗證前置、request / response wrapping、編碼、trace id 等低層 HTTP 處理
  • `HandlerInterceptor` 是 Spring MVC 層級，已經進入 Spring DispatcherServlet，可在 controller 前後做處理，適合登入狀態檢查、controller 層級 logging、locale、簡單權限檢查
  • `AOP` 是 Spring Bean method 層級，透過 proxy 攔截方法呼叫，適合 transaction、service method logging、audit、metrics、權限註解等業務方法橫切關注點
  • 執行順序大致是：Filter → DispatcherServlet → Interceptor → Controller → Service / AOP proxy → 回程 Interceptor → 回程 Filter
  • AOP 要注意 proxy 限制：同類別自呼叫通常不會觸發 advice，非 Spring 管理物件也不會被攔截，private method 也不是典型 AOP 切點
  • 此作業規模若要加 request logging 或 trace id，可用 Filter；若要統一記錄 controller 耗時可用 Interceptor；若要記錄 service 業務操作或交易邊界，才考慮 AOP

建議使用方式：面試時先讓候選人回答「會怎麼做」，再追問「還有沒有其他做法」與「以這份作業規模你會選哪一種」。能說出取捨，比單純背出框架名稱更有鑑別度。
